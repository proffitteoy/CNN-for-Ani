# 冷启动结论

## 项目事实

- 当前仓库初始状态只有通用模板和两份实验文档，没有 Git 元数据、源码、依赖文件、真实验证码、Animeko 采集入口或 Kotlin 工程。
- `实现过程.md` 明确给出采集、标注与 5 折划分、训练、导出与跨语言对齐四个阶段，是实验主执行规范。
- `识别模型.md` 明确任务是固定四位数字的 multi-head classification，输入为 `1 x 32 x 96`，基线为三层小型 CNN，输出为 `4 x 10`。
- 当前机器有 `uv 0.11.5`；可复用的运行时为 Python 3.12.13，但初始环境没有 PyTorch、pytest 或 ruff。

## 已生成的项目契约

- 根目录 `AGENTS.md` 已从通用模板裁剪为本项目操作手册，固定了文档优先级、工具链、目录边界、模型/数据红线和验证要求。
- 根目录 `README.md` 记录真实启动命令、数据命名、实验阶段、当前状态和阻塞项；`docs/README.md` 已裁剪为项目文档索引。
- `pyproject.toml` 固化 Python 3.12、PyTorch、Pillow、NumPy、pytest 和 ruff。
- `src/cnn_for_ani/` 落地了模型、确定性预处理、标签解析、指标和置信度的最小参考实现。
- `tests/` 用不依赖真实验证码的测试保护工程契约；这些测试只证明实现一致性，不代表模型准确率。

## 从模板中删除或改写的内容

- 删除了与 Web、数据库、API、部署、前端状态管理等无事实依据的通用约束。
- 没有套用 Monorepo、服务层、仓储层等通用目录模板。
- 将抽象的“启动/测试/构建”改为本项目实际可执行的 uv、pytest 和 ruff 命令；当前项目没有数据库迁移或部署命令。
- 将泛化的安全输入清单改为该实验真正的风险：原始数据不可变、人工标签不可由模型猜测、批次泄漏、预处理漂移和 Python/Kotlin 数值偏差。

## 当前验证边界

冷启动能验证模型输入输出形状、参数量、标签解析、预处理范围、三项指标和最小位置置信度。它不能验证真实数据分布、5 折准确率、98%/99% 目标、权重导出格式或 Kotlin logits 对齐。

## 下一阶段输入

继续第一阶段前，需要提供 Animeko 源码位置或明确的验证码请求入口。采集时还需确定请求是否合法、采样频率和存储位置；没有这些输入时，不应编造下载脚本或对外部服务进行批量请求。
