# 冷启动

> 用途：拿到一个新项目后，把“通用模板”裁剪成该项目可执行的契约，并在必要时把契约落到仓库本身。

## 目标

- 不直接照搬通用模板。
- 先从仓库事实出发，再形成项目级 `AGENTS.md`。
- 让契约不仅停留在提示词里，而是尽量同步到仓库文档、脚本说明、测试说明和目录边界。

## 一、冷启动输出物

完成一次项目冷启动后，至少应产出以下内容中的一部分：

1. 项目级 `AGENTS.md`
2. 补齐或修正后的 `README.md` / `docs/`
3. 与契约一致的命令说明、配置说明、测试说明
4. 一份冷启动结论，说明哪些规则被保留、删除或改写

注意：

- 输出物以最小必要为原则，不为了“完整”而制造文档噪音。
- 如果仓库已有成熟契约，只需补足缺口，不要推翻重写。

## 二、进入项目后的第一轮盘点

在改任何代码或文档前，先确认以下事实：

### 1. 仓库基础信息

- 是否已有 `AGENTS.md`
- 是否已有 `README.md`、`docs/`、贡献指南、架构说明
- 是单体、Monorepo、库、CLI、Web 应用、数据项目，还是混合仓库

### 2. 技术栈与工具链

- 主语言与版本
- 包管理器和依赖文件
- 启动命令
- 构建命令
- 测试命令
- Lint / format / typecheck 命令
- 数据库迁移、种子数据、部署脚本位置

### 3. 代码组织事实

- 主要源码目录在哪
- 测试目录在哪
- 配置目录在哪
- 静态资源、脚本、生成物、运行时目录在哪
- 核心模块边界是否清晰

### 4. 现有约定与红线

- 已有命名风格
- 错误处理模式
- 状态管理方式
- 是否有不可触碰的第三方集成或本地依赖
- 是否有历史原因导致的特殊目录和脚本

## 三、把通用模板裁剪成项目版 AGENTS

生成项目级 `AGENTS.md` 时，必须做“裁剪”，不能机械复制。

### 应保留的内容

- 回答语言
- 规则优先级
- 修改前先读上下文
- 最小可验证修改
- 文档/测试同步
- 结果汇报要求

### 应按项目改写的内容

- 技术栈与命令入口
- 目录边界和模块职责
- 本项目的启动 / 测试 / 构建 / 发布方式
- 本项目常见风险点
- 哪些类型的变更必须补测试
- 哪些专项任务需要查看哪些 skill 或模板

### 应删除的内容

- 与当前项目无关的语言或框架说明
- 没有证据支持的目录结构建议
- 不适用于该仓库的强约束
- 会与现有团队习惯冲突但无明确收益的条款

## 四、如何把契约落到仓库

项目级 `AGENTS.md` 不是终点。若仓库事实与契约不一致，应在最小风险前提下同步仓库。

### 适合直接落地的修改

- `README.md` 缺少真实启动、测试、构建命令时，补上
- `docs/` 缺少关键架构或目录说明时，补最短必要文档
- `.env.example` 缺少关键变量说明时，补上示例
- 已存在脚本但文档没写时，补说明
- 测试入口存在但没人知道怎么跑时，补命令说明
- 目录命名、路径说明与文档明显不一致时，修正文档

### 不适合在冷启动阶段贸然做的事

- 大规模重构目录
- 替换主要框架或基础设施
- 凭主观偏好重写仓库风格
- 为了套模板而新增一堆并不会被使用的文件
- 在没证据的情况下新增“标准命令”或“标准目录”

## 五、skills 与专项模板的使用策略

冷启动阶段要顺手决定：这个项目以后哪些任务该看哪些专项文档。

建议在项目级 `AGENTS.md` 中明确：

- 普通代码修复、接口开发、测试补充：默认不看专项 skill
- 前端视觉设计：需要时看对应设计类 skill
- PDF / DOCX / PPTX / 表格类任务：按文件类型命中对应 skill
- 安全审计、权限边界梳理：看 [代码审计.md](./代码审计.md)
- 结构散乱、准备补边界：按需参考 [代码组织.md](./代码组织.md)

核心原则：

- skill 是补充，不是顶层规则
- 项目级 `AGENTS.md` 才是该仓库的直接操作手册

## 六、推荐执行顺序

1. 读仓库现有 `AGENTS.md`、`README.md`、`docs/`
2. 读构建与依赖文件，确认真实命令
3. 看关键代码目录和测试目录，理解模块边界
4. 归纳项目特有规则与风险
5. 基于 [AGENTS.md](./AGENTS.md) 生成项目级版本
6. 修正仓库里与契约不一致但能低风险落地的文档或说明
7. 用最小验证确认文档和命令没有明显虚构

## 七、冷启动检查清单

- [ ] 我是否先基于仓库事实，而不是先套模板
- [ ] 我是否确认了真实的启动、测试、构建命令
- [ ] 我是否识别了主要源码目录、测试目录和配置目录
- [ ] 我是否把不适用的模板内容删掉了
- [ ] 我是否避免了无关重构和伪标准化
- [ ] 我是否同步更新了 README / docs / 示例配置中最关键的缺口
- [ ] 我是否明确了这个项目以后何时需要看 skills

## 八、推荐交付格式

完成冷启动后，建议在最终说明里按下面结构汇报：

1. 项目事实
2. 生成了哪些项目级契约
3. 为了让契约落地，顺手修改了哪些仓库文件
4. 如何验证这些契约不是空话
5. 剩余风险和后续建议
