# CNN for Ani 项目协作约定

## 1. 项目定位与规则优先级

- 默认使用中文沟通。
- 本项目用于训练固定长度四位数字验证码识别模型，并为后续 Animeko 中的纯 Kotlin 前向推理产出可复现的模型契约。
- 规则优先级：用户当前要求 > 本文件 > `docs/实现过程.md` > `docs/识别模型.md` > 根目录 `README.md`、`pyproject.toml`、现有代码与测试 > `docs/` 中的通用模板。
- `docs/实现过程.md` 是完整实验的主执行规范；`docs/识别模型.md` 是模型、损失、指标、增强和置信度设计的理论基础。两者冲突时，先停止实现并说明冲突，不自行发明第三套方案。

## 2. 技术栈与真实命令

- Python 3.12，使用 `uv` 管理环境和依赖。
- PyTorch 负责训练和 Python 参考前向；Pillow/NumPy 负责可在 Kotlin 端复现的简单预处理。
- 安装：`uv sync --python 3.12`
- 冒烟检查：`uv run python -m cnn_for_ani`
- 测试：`uv run pytest`
- 静态检查：`uv run ruff check .`
- 格式检查：`uv run ruff format --check .`

## 3. 目录边界

- `src/cnn_for_ani/`：模型、预处理、数据契约、指标和后续训练/导出逻辑。
- `tests/`：与 Python 模型契约、数据契约以及未来 Python/Kotlin logits 对齐相关的测试。
- `dataset/raw/`：采集到的原始验证码，只允许新增和读取，不得覆盖、变换或删除。
- `dataset/labeled/`：人工确认标签后的样本，命名为 `<四位数字>_<样本ID>.<扩展名>`。
- `artifacts/`：训练检查点、报告和最终 `captcha.weights`，默认不提交版本库。
- `docs/`：实验规范、理论依据和冷启动结论。

## 4. 模型与数据硬约束

- 任务固定为四位数字的 multi-head classification，不引入字符分割、CRNN、CTC、Transformer 或通用 OCR 流程，除非真实数据证明当前假设失败且用户确认变更方向。
- 输入固定为 `1 x 32 x 96` 灰度图，像素转换为 `float32 [0, 1]`；不得擅自加入 ImageNet normalization。
- 基线模型是 3 层 stride-2 小型 CNN，输出形状必须为 `[B, 4, 10]`。
- 损失是四个位置交叉熵的平均值。至少汇报 CharAcc、ExactAcc 和 HammingError，主要决策指标是 ExactAcc。
- 置信度取四个位置最大类别概率的最小值，不使用平均置信度掩盖单个低置信位置。
- 可用增强仅限小幅平移、亮度、对比度、小角度旋转和受控噪声；禁止水平/垂直翻转、大角度旋转和随机裁剪。
- 数据按下载批次存在时优先 Group K-Fold，严禁同批高度相似样本跨训练集与验证集造成泄漏。
- 原始样本和元数据属于实验事实。不得自动猜标签，不得把推理结果回写为人工标签。

## 5. 变更与验证要求

- 优先最小可验证修改，不为了套模板增加当前阶段不会使用的基础设施。
- 改模型结构、输入预处理、标签解析、权重顺序或二进制格式时必须补测试，并检查 Python/Kotlin 复现影响。
- 训练逻辑变更至少验证输出形状、损失可反传和三项指标；数据划分变更至少验证互斥、完整覆盖、确定性和批次隔离。
- 最终 Kotlin 实现必须与 Python 对同一输入的 40 个 logits 对齐，最大绝对误差小于 `1e-4`；只比较 argmax 不算通过。
- 每次修改后检查 README、实验文档、测试和示例命令是否需要同步。
- 当前没有真实验证码数据、Animeko 采集入口或 Kotlin 工程时，必须明确标记相关验证为未执行，不得用合成数据替代真实准确率结论。

## 6. skills 使用策略

- 普通 Python 模型、数据和测试修改默认遵守本文件与实验文档，不额外叠加专项 skill。
- 安全或信任边界审查按需参考 `docs/代码审计.md`。
- 目录边界确实需要调整时才参考 `docs/代码组织.md`，不能覆盖本项目已确定的实验边界。

## 7. 交付说明

最终回复应说明：修改内容、依据的实验约束、实际运行的验证、未验证部分和下一阶段所需输入。涉及实验结果时，区分“代码冒烟通过”“合成数据验证”和“真实数据结论”。
